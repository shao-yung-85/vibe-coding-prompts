# 🤝 貢獻指南 Contributing Guide

感謝你想貢獻！這個專案的價值在於**每個 prompt 都真的好用，而且解釋得清楚**。請花兩分鐘讀完這份指南，讓你的 PR 能順利合併。

---

## 🌟 核心原則：寧缺勿濫

我們只收符合這兩個條件的 prompt：

1. ✅ **可以直接複製貼上就用** — 不是抽象描述，是真正能丟進 AI 工具的成品。
2. ✅ **附上「為什麼有效」** — 解釋背後的原理，讓使用者能理解並改成自己的版本。

湊數的、沒解釋的、跟現有模板高度重複的，會被婉拒。一個好 prompt 勝過十個普通 prompt。

---

## 🚀 貢獻流程

1. **Fork** 這個 repo
2. 開一個分支：`git checkout -b add/your-prompt-name`
3. 找到對應分類，依照下方**格式規範**新增你的 prompt
4. 自己先測一次：把 prompt 複製貼到 AI 工具，確認它真的有用
5. Commit：`git commit -m "Add: [分類] 你的 prompt 簡述"`
6. 發 **Pull Request**，在描述裡簡單說明這個 prompt 解決什麼問題

---

## 📐 Prompt 格式規範

每個 prompt 模板**必須**遵循以下四段式格式。直接照抄這個模板來填：

````markdown
### N. 簡短有力的標題（看標題就知道用途）

```
[Prompt 本體]
這裡放可以「直接複製貼上」的完整 prompt。
需要使用者填空的地方，用 [方括號] 標示，例如 [功能描述]、[貼上程式碼]。
語氣自然、像在跟 AI 對話，不要寫成抽象的規格說明。
```

**為什麼有效：** 用 2-4 句話解釋這個 prompt 背後的原理——它觸發了 AI 的什麼行為、避開了什麼常見陷阱、為什麼能產出好結果。這是必填，不能省略。
````

### ✍️ 各部分的要求

| 部分 | 要求 |
|------|------|
| **標題** | 一句話講清楚用途，加上編號。看標題就知道什麼時候該用它 |
| **Prompt 本體** | 放在 ` ``` ` 程式碼區塊裡（方便一鍵複製）。要填的地方用 `[方括號]`。必須是完整、可直接用的內容 |
| **為什麼有效** | 2-4 句，解釋原理。**這是這個專案的靈魂，絕對不能少** |

## ✅ 好範例 vs ❌ 壞範例

### ✅ 好範例

````markdown
### 3. 先診斷，不要急著改

```
我遇到這個錯誤：
[完整錯誤訊息]

先不要改任何程式碼。請你：
1. 解釋這個錯誤實際代表什麼
2. 列出 2-3 個最可能的原因
3. 告訴我要怎麼確認是哪一個
```

**為什麼有效：** AI 看到錯誤的直覺是「改點東西試試」，常常治標不治本還引入新 bug。強制它先分析、提假設、給驗證方法，能命中真正的根因。
````

👆 好在哪：標題清楚、prompt 完整可貼、有填空標示、解釋講到原理。

### ❌ 壞範例

````markdown
### Debug 用的

```
幫我修 bug
```
````

👆 壞在哪：標題含糊、prompt 太短沒實際內容、沒有填空引導、**完全沒有「為什麼有效」**。這種會被退回。

## 📋 分類對照

請把 prompt 加到最貼切的分類。現有分類：

- 🚀 專案初始化
- ✨ 新增功能
- 🐛 Debug 除錯
- ♻️ 重構程式碼
- 🧪 寫測試
- 🎨 UI / 樣式
- 🛡️ 護欄 Prompt（不讓 AI 亂動）
- 📋 CLAUDE.md 模板

覺得需要**新分類**？沒問題，但請在 PR 描述裡說明為什麼現有分類放不下，並至少先準備 3 個該分類的 prompt。

---

## 🎨 風格要求

- **實用優先，零廢話** — 不要寫長篇大論的前言
- **中文為主** — 說明用繁體中文，prompt 本體可中可英（看實際使用情境）
- **適度 emoji** — 用來分區、標重點，但不要每行都有
- **保持一致** — 新增的內容請對齊現有模板的語氣和排版

---

## 🐛 其他貢獻方式

不一定要加新 prompt 才能貢獻：

- 修正錯字、語句不通順
- 改善現有 prompt 的「為什麼有效」說明
- 補充工具對應表的資訊
- 翻譯 / 改善英文摘要

這些都很歡迎，直接發 PR 即可。
