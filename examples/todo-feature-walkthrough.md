# 🚶 實戰走查：從零做一個待辦清單功能

這個範例示範**怎麼把多個 prompt 串起來**完成一件事。重點不是每個 prompt 本身（那些在主 README 都有），而是**順序、銜接、以及在哪裡踩煞車**。

情境：用 Cursor 或 Claude Code，做一個簡單的待辦清單 Web App。

---

## 第 0 步：先講清楚，別急著寫

用 [🚀 專案初始化 #1](../README.md#-專案初始化)：

```
我要做一個待辦清單 Web App。
技術棧：React + TypeScript + Vite，資料先存 localStorage 就好。
目標使用者：想快速記事情的個人。

先不要寫任何程式碼。請你：
1. 用條列方式重述你理解的需求
2. 列出你建議的資料夾結構
3. 指出你覺得有疑慮或需要我決定的地方

我確認後你再開始動工。
```

> 💬 **為什麼從這裡開始：** 90% 的「AI 做歪了」都源自第一步沒對齊。先讓它重述、列結構、提疑慮，你花 2 分鐘就能擋掉後面 2 小時的返工。

AI 回完後，你可能發現它假設了「要支援多個清單」，但你只想要一個。當場糾正，再往下。

---

## 第 1 步：先要一個能跑的骨架

用 [🚀 專案初始化 #2](../README.md#-專案初始化)：

```
請建立這個專案能跑起來的最小骨架：
- 只要能成功啟動 + 顯示一個 "Hello" 畫面就好
- 包含必要的設定檔
- 不要加任何業務邏輯、不要先寫功能

完成後告訴我怎麼啟動，我跑起來確認沒問題再繼續。
```

> 💬 **為什麼不直接做功能：** 先確認「能跑」，建立一個已知正常的基準點。之後每加一個東西，如果壞了，你會知道是新加的那塊的問題。

`npm run dev` 跑起來看到畫面 → 過關，繼續。

---

## 第 2 步：加第一個功能（只做一條路徑）

用 [✨ 新增功能 #2 垂直切片](../README.md#-新增功能)：

```
請只實作這個功能的最小垂直切片：
- 從「在輸入框打字 → 按 Enter → 待辦項目出現在清單上」的一條完整路徑
- 刪除、編輯、勾選完成、樣式美化先不要做
- 先讓「新增」這條主流程能 work

完成後我測試，沒問題我們再加細節。
```

> 💬 **為什麼一次只做一條：** 「做完整的待辦功能」會生出一大坨沒測過的程式碼。先打通「新增」這一條，驗證資料流對了，再分批加刪除、勾選。

---

## 第 3 步：撞到 bug，先別讓它亂改

假設你發現重新整理頁面後待辦項目都不見了。用 [🐛 Debug #1 先診斷](../README.md#-debug-除錯)：

```
我遇到這個問題：

我預期：重新整理頁面後，待辦項目還在
實際發生：重新整理後清單變空的

先不要改任何程式碼。請你：
1. 解釋這個問題實際代表什麼
2. 列出 2-3 個最可能的原因，由最可能排到最不可能
3. 告訴我要怎麼確認是哪一個

我們確認根因後再修。
```

> 💬 **為什麼先診斷：** 如果直接說「修好它」，AI 可能會亂加一堆 useEffect 試試看。先讓它分析，它會說：「最可能是沒把資料寫進 localStorage，或讀取時機不對」——命中根因，一次修好。

---

## 第 4 步：功能差不多了，補測試把它鎖住

用 [🧪 寫測試 #1 涵蓋邊界](../README.md#-寫測試)：

```
請為「新增待辦」這個功能寫測試。

除了正常情況，請務必涵蓋：
- 邊界值（空字串、只有空白、超長文字）
- 重複新增同樣的內容
- 特殊字元

每個測試用清楚的名稱說明它在驗證什麼。用 Vitest 撰寫。
```

> 💬 **為什麼這時候才寫測試：** 功能穩定了再補測試，把「現在的正確行為」鎖住。之後重構或加功能時，測試會幫你看著有沒有改壞。

---

## 第 5 步：要重構，但畫一條紅線

程式碼有點亂了，想整理。用 [♻️ 重構 #1 行為不變](../README.md#-重構程式碼) + [🛡️ 護欄 #3 最小改動](../README.md#️-護欄-prompt不讓-ai-亂動)：

```
請重構 TodoList 元件，把它拆得更清楚。

絕對規則：
- 對外行為必須完全不變
- 不要新增功能、不要改 API
- 剛剛寫的測試必須全部持續通過
- 只改結構與命名，不要「順便」優化別的東西

改完請說明你做了哪些改動，以及如何確認行為沒變。
```

> 💬 **為什麼疊兩個 prompt：** 「行為不變」管重構安全，「最小改動」管它別越界。有第 4 步的測試當安全網，重構後跑一次測試就知道有沒有改壞。

---

## 收尾：把規則固定下來

如果你會持續開發這個專案，把上面反覆用到的護欄寫進 `CLAUDE.md`，之後就不用每次重貼。直接抄 [CLAUDE-md-template.md](../CLAUDE-md-template.md) 的精簡版即可。

## 🎯 這個走查的重點整理

| 階段 | 心法 | 用到的護欄 |
|------|------|-----------|
| 開始 | 先對齊需求，別急著寫 | 規劃確認閘門 |
| 建骨架 | 先要「能跑」的基準點 | — |
| 加功能 | 一次只做一條垂直路徑 | 最小改動 |
| Debug | 先診斷根因，別亂改 | 先診斷不要急著改 |
| 寫測試 | 功能穩了再鎖住行為 | — |
| 重構 | 行為不變 + 有測試保護 | 最小改動、不破壞 |

**一句話總結：** 好的 vibe coding 不是「下一個神 prompt 一次到位」，而是**小步前進、每步驗證、隨時能踩煞車**。
