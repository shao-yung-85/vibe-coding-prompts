## 🤔 這個專案解決什麼問題？

用 AI 寫程式（vibe coding）時，最常遇到三件事：

1. **不知道怎麼開口** — 對著輸入框發呆，prompt 寫得太模糊，AI 給的東西不能用。
2. **AI 亂改** — 你只想加一個按鈕，它順手重構了半個專案、刪掉你的測試、改了不相關的檔案。
3. **結果不可控** — 同一個需求，運氣好一次到位，運氣不好來回十輪還在繞。

這個 repo 把「會穩定產出好結果的 prompt」整理成可複製貼上的模板，**而且每一個都解釋為什麼有效**——因為知道原理，你才能改成自己的版本。

## 🚀 專案初始化

開新專案時，第一個 prompt 決定了後面所有對話的地基。地基歪了，後面全部歪。

### 1. 先講清楚再寫一行 code

```
我要做一個 [專案的一句話描述]。
技術棧：[語言 / 框架 / 資料庫]。
目標使用者：[誰會用]。

先不要寫任何程式碼。請你：
1. 用條列方式重述你理解的需求
2. 列出你建議的資料夾結構
3. 指出你覺得有疑慮或需要我決定的地方

我確認後你再開始動工。
```

**為什麼有效：** 強制 AI 進入「規劃模式」而不是「立刻生成模式」。讓它重述需求能在第一輪就抓出誤解；要求它列出疑慮，能逼出那些它原本會擅自決定的隱藏假設。

### 2. 最小可跑骨架（Walking Skeleton）

```
請建立這個專案能跑起來的最小骨架：
- 只要能成功啟動 + 顯示一個 "Hello" 畫面就好
- 包含必要的設定檔（package.json / requirements.txt 等）
- 不要加任何業務邏輯、不要先寫功能

完成後告訴我怎麼啟動，我跑起來確認沒問題再繼續。
```

**為什麼有效：** 一次生成整個專案幾乎一定跑不起來，而且很難 debug。先要一個「能跑」的最小版本，建立一個已知可運作的基準點，之後每加一個功能都站在穩定地基上。

### 3. 鎖定版本與慣例

```
在開始之前，請依照以下約束建立專案：
- [語言] 版本：[x.x]
- 套件管理工具：[npm / pnpm / pip / poetry]
- 程式風格：[ESLint + Prettier / Black / ...]
- 不要使用已棄用（deprecated）的 API

如果你打算用某個我沒提到的主要套件，先問我再裝。
```

**為什麼有效：** AI 的訓練資料橫跨多個版本，預設可能混用新舊語法。明確鎖版本與工具，避免生出一個你本機跑不動、或半新半舊的縫合怪。

### 4. 一次決定資料夾結構

```
請提出 2 種這個專案的資料夾結構方案：
- 方案 A：適合快速開發、檔案少
- 方案 B：適合長期維護、模組清楚

各列出優缺點與適用情境，並推薦一個。我選定後請建立空的資料夾與佔位檔。
```

**為什麼有效：** 結構在中途才改最痛苦（一堆 import 路徑要動）。讓 AI 先給選項並說明取捨，你能在零成本時做好這個高槓桿決定。

### 5. 建立 README 與開發指令

```
請為這個專案產生一份 README.md，包含：
- 一句話專案簡介
- 環境需求
- 安裝步驟（可複製貼上的指令）
- 如何啟動 / 如何測試
- 資料夾結構說明

指令請用實際可執行的形式，不要寫 "安裝相依套件" 這種模糊描述。
```

**為什麼有效：** 逼 AI 把「怎麼跑這個專案」明確寫下來，等於做了一次健全性檢查——如果它寫不出可執行的指令，代表專案設定本身就有問題。

## ✨ 新增功能

加功能最容易讓 AI 失控，因為它會「順便」碰它覺得相關的東西。關鍵是縮小範圍、講清楚邊界。

### 1. 先計畫，後實作

```
我想新增功能：[功能描述]。

先不要寫程式碼。請你：
1. 列出會新增 / 修改哪些檔案
2. 說明每個檔案大概要改什麼
3. 指出有沒有會影響到現有功能的地方

我看過計畫、回覆 OK 之後你再開始寫。
```

**為什麼有效：** 把「規劃」和「執行」拆成兩步。你在便宜的階段（還沒寫 code）就能攔截錯誤方向，也能預先看到它打算動哪些檔案，防止它亂擴張範圍。

### 2. 垂直切片（一次只做一條完整路徑）

```
請只實作這個功能的最小垂直切片：
- 從 [使用者操作] 到 [看到結果] 的一條完整路徑
- 其他邊界情況、錯誤處理、樣式美化先不要做
- 先讓主流程能 work

完成後我測試，沒問題我們再加細節。
```

**為什麼有效：** 「一次做完整功能」會產生大量未經測試的程式碼。先打通一條最細的端到端路徑，能快速驗證設計可行，後續再分批補強，每步都可驗證。

### 3. 沿用既有模式

```
請參考專案中現有的 [某個類似功能 / 檔案，例如 src/features/login] 的寫法，
用同樣的結構與風格來實作 [新功能]。

保持與現有程式碼一致的命名、資料夾位置和錯誤處理方式。
```

**為什麼有效：** AI 預設會用「它認為最好」的寫法，導致同一個專案裡風格混亂。給它一個現成範例當錨點，產出的程式碼會自然融入專案，而不是格格不入。

### 4. 明確列出驗收條件

```
新增功能：[描述]。

完成的定義（必須全部滿足）：
- [ ] [條件 1，例如：輸入 email 格式錯誤時顯示提示]
- [ ] [條件 2，例如：成功後導向首頁]
- [ ] [條件 3，例如：載入時顯示 loading]

請逐項實作，完成後對照清單告訴我每一項的狀態。
```

**為什麼有效：** 模糊的需求換來模糊的結果。把「完成」拆成可勾選的具體條件，AI 有明確標靶，你也有客觀的驗收依據，不會出現「它覺得做完了但其實少一半」。

### 5. 整合到既有流程

```
我要把 [新功能] 接到現有的 [某流程，例如結帳流程]。

請先讀過相關的現有檔案，告訴我：
1. 接入點在哪裡（哪個函式 / 元件）
2. 需要傳遞什麼資料
3. 會不會破壞現有行為

確認後再修改，並盡量不更動現有函式的對外介面。
```

**為什麼有效：** 新功能最大的風險是破壞舊功能。要求 AI 先「讀懂」再「接入」，並保護現有介面不變，能大幅降低改 A 壞 B 的連鎖傷害。

## 🐛 Debug 除錯

Debug 時最忌諱讓 AI 亂猜亂改。好的 prompt 會強迫它先找根因，再動手。

### 1. 先診斷，不要急著改

```
我遇到這個錯誤：
[完整錯誤訊息 / stack trace]

重現步驟：
[1. ... 2. ... 3. ...]

先不要改任何程式碼。請你：
1. 解釋這個錯誤實際代表什麼
2. 列出 2-3 個最可能的原因，由最可能排到最不可能
3. 告訴我要怎麼確認是哪一個

我們確認根因後再修。
```

**為什麼有效：** AI 看到錯誤的直覺反應是「改點東西試試看」，常常治標不治本還引入新 bug。強制它先分析、提假設、給驗證方法，能命中真正的根因。

### 2. 提供完整脈絡

```
這段程式碼的行為不如預期。

我預期：[預期行為]
實際發生：[實際行為]

相關程式碼：
[貼上程式碼]

相關的輸入 / 資料：
[貼上實際資料]

請找出為什麼預期和實際不一致。
```

**為什麼有效：** 「預期 vs 實際」是 debug 的黃金格式。明確的落差描述加上真實資料，讓 AI 不用猜你想要什麼，能直接聚焦在差異本身。

### 3. 加 log 縮小範圍

```
我不確定問題出在哪一段。請在這段流程的關鍵位置加上 console.log / print，
讓我能看到每一步的變數值。

先不要改邏輯，只加觀測用的輸出。我跑完把結果貼給你，我們再一起判斷。
```

**為什麼有效：** 當根因不明時，與其讓 AI 瞎改，不如先增加可觀測性。用真實的執行資料縮小範圍，比任何猜測都準，也避免在錯誤的地方動刀。

### 4. 二分法定位

```
這個功能昨天還正常，今天壞了。我不確定是哪個改動造成的。

請用二分法協助我定位：
- 告訴我可以先檢查 / 暫時停用哪一部分
- 根據結果再縮小範圍

一次只給我一個明確的檢查動作。
```

**為什麼有效：** 把大海撈針變成有系統的對半搜尋。一次只做一個動作避免變因混淆，能在最少步數內鎖定出問題的那段改動。

### 5. 解釋這段我看不懂的程式碼

```
我不理解這段程式碼為什麼會這樣運作（或為什麼會出錯）：
[貼上程式碼]

請逐行解釋它實際在做什麼，特別標出：
- 任何有副作用的地方
- 任何可能在邊界情況出錯的地方

用我這個 [初學者 / 中級] 程度聽得懂的方式說明。
```

**為什麼有效：** 很多 bug 來自「你以為這段在做 A，其實它在做 B」。讓 AI 當翻譯，逐行還原真實行為並點出副作用與邊界風險，常常解釋到一半你自己就發現問題了。

## ♻️ 重構程式碼

重構是 AI 最容易「越幫越忙」的場景——它會順手改掉行為。重點是：**只改結構，不改行為**。

### 1. 行為不變的重構

```
請重構這段程式碼：
[貼上程式碼]

絕對規則：
- 對外行為必須完全不變（輸入輸出一致）
- 不要新增功能、不要改 API
- 只改善可讀性 / 結構 / 命名

改完請說明你做了哪些改動、為什麼，以及如何確認行為沒變。
```

**為什麼有效：** 「行為不變」是重構的鐵律。把它寫死成絕對規則，能擋住 AI 趁機「順便優化功能」的衝動，讓你能安心地只審查結構變化。

### 2. 先有測試再重構

```
我想重構 [某段程式碼]，但擔心改壞。

第一步：請先為它目前的行為寫測試（涵蓋主要情況與邊界），確保現在是綠的。
第二步：等我確認測試通過後，再進行重構，並讓測試持續通過。

先只做第一步。
```

**為什麼有效：** 沒有測試的重構等於閉著眼睛動手術。先用測試把「現在的正確行為」鎖住，重構後測試還綠就代表行為沒變——這是最安全的重構工作流。

### 3. 一次只做一種重構

```
這段程式碼有多個可以改善的地方。請一次只做一種重構，分批進行。

這一輪：只做 [例如：把重複的程式碼抽成函式]，其他先不要碰。

完成後我 review，再進行下一種。
```

**為什麼有效：** 一次混合多種重構（改名 + 抽函式 + 換結構）會讓 diff 變得無法 review，也難以判斷哪個改動出了問題。分批進行讓每個 diff 小而聚焦。

### 4. 消除重複（DRY）

```
我覺得這幾個地方有重複的邏輯：
[貼上幾段相似的程式碼]

請判斷它們是不是真的可以合併。如果可以，抽成共用的函式 / 模組；
如果它們只是長得像但本質不同，請直接告訴我不該合併，並說明原因。
```

**為什麼有效：** 主動給 AI「可以說不」的選項，避免它為了消除重複而強行合併本質不同的邏輯（這會製造出難用的過度抽象）。讓它做判斷，而不是無腦套用規則。

### 5. 拆分過大的函式 / 元件

```
這個 [函式 / 元件] 太大、做太多事了：
[貼上程式碼]

請把它拆成幾個職責單一的小單位：
- 每個單位只做一件事
- 命名要能看出它負責什麼
- 保持原本的對外行為不變

先給我拆分計畫（要拆成哪幾塊），我同意後再動手。
```

**為什麼有效：** 「先給計畫」讓你能在實作前評估拆分方式是否合理。要求「職責單一 + 命名清楚」給了明確的拆分標準，避免拆得支離破碎或拆得沒意義。

## 🧪 寫測試

AI 寫測試的通病是「只測 happy path」或「為了綠而綠」。好 prompt 會要求它測對地方。

### 1. 涵蓋邊界與錯誤情況

```
請為這個函式寫測試：
[貼上程式碼]

除了正常情況，請務必涵蓋：
- 邊界值（空值、0、負數、超大值、空字串 / 空陣列）
- 錯誤輸入（型別錯誤、格式錯誤）
- 例外處理是否如預期

每個測試用清楚的名稱說明它在驗證什麼。
```

**為什麼有效：** Bug 幾乎都躲在邊界與錯誤路徑，而不是正常流程。明確列出要涵蓋的情況，避免 AI 只寫三個 happy path 測試就交差。

### 2. 測行為，不要測實作細節

```
請為 [某功能] 寫測試。

原則：測「對外可觀察的行為與結果」，不要測內部實作細節。
這樣之後重構內部時，測試不應該因此壞掉。

用 [測試框架] 撰寫。
```

**為什麼有效：** 綁死實作細節的測試很脆弱，重構時一改就紅，最後大家乾脆刪測試。聚焦在行為的測試才有長期價值，能保護你重構時不出錯。

### 3. 從測試案例清單開始

```
我要為 [功能] 寫測試。

先不要寫程式碼。請先列出你打算寫的測試案例清單（用一句話描述每個案例在驗證什麼），
分成「正常情況 / 邊界情況 / 錯誤情況」三類。

我看過清單、補充或刪減後，你再寫實際的測試碼。
```

**為什麼有效：** 先審清單能在零成本時發現「漏測的情況」或「多餘的測試」。等清單對了再寫 code，比寫完一堆測試才發現方向錯了有效率得多。

### 4. 重現 bug 的回歸測試

```
我們剛修好一個 bug：[描述 bug]。

請寫一個回歸測試：
- 這個測試在修復前應該會失敗（重現 bug）
- 在修復後應該通過

這樣未來如果有人不小心改回去，測試會立刻抓到。
```

**為什麼有效：** 每個修好的 bug 都該留下一個「守門員」。這種測試把一次性的修復變成永久的保護，防止同樣的問題以後再次溜回來。

### 5. 為既有程式碼補測試（建立安全網）

```
這段程式碼還沒有任何測試，我也還不打算改它的行為：
[貼上程式碼]

請為它「目前的行為」補上測試（characterization tests），
忠實記錄它現在實際的輸出——即使某些行為看起來怪怪的，也先照實寫下來，
並把你覺得可疑的地方標記出來給我看。
```

**為什麼有效：** 面對沒測試的舊程式碼，第一步不是修正而是「鎖定現狀」。先把現有行為用測試固定住，建立安全網，之後才能放心重構或修 bug。

## 🎨 UI / 樣式

UI 是最吃「具體描述」的場景。越模糊（「弄好看一點」），結果越不可控。

### 1. 給參考、給約束

```
請幫我做一個 [元件，例如：定價卡片] 的 UI。

風格參考：[例如：類似 Stripe 的簡潔現代風 / 貼一張截圖描述]
約束：
- 使用我們現有的 [CSS 框架 / 設計系統]
- 配色用：[主色 / 輔色]
- 要支援 RWD（手機 + 桌機）

先給我一個版本，我再提調整。
```

**為什麼有效：** 「好看」對每個人定義不同。給具體的風格參考和硬性約束（框架、配色、RWD），把抽象審美轉成可執行的規格，AI 才打得中你要的方向。

### 2. 只改樣式，不動邏輯

```
請只調整這個元件的「外觀樣式」：
[貼上程式碼]

需求：[例如：間距加大、按鈕改圓角、字級調整]

絕對不要動任何邏輯、state、事件處理或資料流，只改 className / style。
```

**為什麼有效：** 調樣式時 AI 常順手「優化」邏輯，反而改出 bug。明確劃出「只准動樣式」的紅線，把改動範圍鎖在視覺層，讓 diff 安全又好審。

### 3. 用設計 token 而非寫死數值

```
請幫這個介面套用樣式，但不要寫死顏色和尺寸的魔術數字。

請使用變數 / 設計 token（例如 CSS variables、Tailwind 的 spacing scale），
讓之後要統一調整時只要改一個地方。如果專案還沒有 token 系統，先建議一套。
```

**為什麼有效：** 寫死的 `#3b82f6`、`margin: 13px` 散落各處會讓後續調整變地獄。用 token 讓樣式系統化、可維護，這是 AI 預設不會做但對長期很重要的事。

### 4. 無障礙（a11y）優先

```
請幫我做 [元件]，並從一開始就符合無障礙標準：
- 正確的語意化 HTML 標籤
- 鍵盤可操作（Tab / Enter / Esc）
- 適當的 aria 屬性與 label
- 足夠的色彩對比

完成後列出你做了哪些無障礙處理。
```

**為什麼有效：** 無障礙事後補很痛苦，從頭就做幾乎零成本。在 prompt 裡明列要求，能把 a11y 變成預設行為，順便讓 AI 用更正確的語意化結構。

### 5. 漸進式調整（小步快跑）

```
這個畫面我想微調，但要一步一步來。

這一輪只改：[一個具體的小調整，例如：把標題置中並加大]

不要連帶改其他東西。改完我看效果，滿意再講下一個調整。
```

**為什麼有效：** UI 調整很主觀，大改一輪常常「這裡好了但那裡又跑掉」。小步快跑讓每次只驗證一個變化，你能精準控制最終長相，也容易回退不滿意的改動。

## 🗄️ 資料庫 / SQL

資料庫改動風險最高——一條 migration 寫錯可能毀掉真實資料。這類 prompt 強調安全與可回復。

### 1. 安全的 Migration（可回滾）

```
我要做這個資料庫變更：[例如：在 users 表加一個 phone 欄位]。

請產生 migration，並務必：
- 同時寫好 up（變更）和 down（回滾）兩個方向
- 新欄位設為 nullable 或給預設值，避免既有資料卡住
- 不要刪除或改名任何現有欄位，除非我明確要求

寫完先給我看 migration 內容，我確認後再執行。
```

**為什麼有效：** 強制「可回滾」和「不破壞既有資料」是資料庫安全的底線。要求先審查再執行，能在不可逆的操作落地前攔下錯誤——資料毀了通常救不回來。

### 2. 解釋並優化這條查詢

```
這條 SQL 查詢很慢：
[貼上 SQL]

請：
1. 解釋它目前實際在做什麼、為什麼可能慢（例如全表掃描、缺索引）
2. 提出優化版本
3. 說明需不需要加索引，加在哪個欄位、為什麼

先不要改線上資料，只給我分析和建議。
```

**為什麼有效：** 先要「解釋為什麼慢」逼 AI 真正分析執行計畫，而不是亂套優化。把「分析」和「執行」分開，讓你能評估建議再決定是否套用到正式環境。

### 3. 防止 SQL Injection

```
請檢查 / 撰寫這段資料庫存取程式碼：
[貼上程式碼]

規則：
- 一律使用參數化查詢（prepared statement / parameterized query）
- 絕對不要用字串拼接組 SQL
- 使用者輸入在進查詢前都視為不可信

如果現有程式碼有拼接 SQL 的地方，標出來並改成參數化。
```

**為什麼有效：** SQL injection 是最常見也最致命的漏洞之一。明確禁止字串拼接、要求參數化，把這條安全鐵律變成 AI 的預設行為，從源頭堵掉注入風險。

### 4. 設計資料表結構

```
我要儲存 [描述資料，例如：訂單與訂單明細]。

請先不要寫 SQL。先提出資料表設計：
- 有哪些表、各自的欄位與型別
- 主鍵、外鍵、索引怎麼設
- 一對多 / 多對多關係怎麼處理

用文字 + 簡單關係圖說明，我確認後你再產生建表語句。
```

**為什麼有效：** 表結構一旦有資料就很難改。先在「設計階段」討論關係與索引，把高槓桿決定提前到零成本時做，避免上線後才發現結構不合理要痛苦遷移。

### 5. 安全地寫資料修正腳本

```
我需要修正正式資料庫裡的一批資料：[描述要改什麼]。

請寫一個資料修正腳本，並且：
- 先寫一段「只 SELECT、不修改」的查詢，讓我確認會影響哪些列、共幾筆
- 修改語句一定要有明確的 WHERE 條件，絕不能漏
- 包在交易（transaction）裡，並告訴我怎麼驗證後再 commit

先給我那段 SELECT 確認查詢。
```

**為什麼有效：** 「先 SELECT 確認再 UPDATE」是資深工程師的肌肉記憶。先看清楚會動到哪些資料、用交易保護、強制 WHERE 條件，能避免「忘了加 WHERE 把整張表改掉」這種災難。

## 🚢 部署 / DevOps

部署環節的 prompt 重點是「可重現」和「別把密鑰外洩」。

### 1. 寫 Dockerfile

```
請為這個專案寫一個 Dockerfile。

需求：
- 基底映像用 [例如：node:20-alpine]，盡量小
- 用多階段建置（multi-stage build）分離建置與執行
- 不要把開發相依、密鑰、.env 打包進去
- 設定非 root 使用者執行

寫完逐行說明每一步在做什麼。
```

**為什麼有效：** 明列「多階段、最小映像、非 root、不含密鑰」這些最佳實務，避免 AI 產出肥大又不安全的預設 Dockerfile。要求逐行說明也讓你看得懂、改得動。

### 2. 設定 CI 流程

```
請幫我設定 [GitHub Actions / GitLab CI] 流程，在每次 push 時：
1. 安裝相依
2. 跑 lint
3. 跑測試
4. （PR 才跑）建置檢查

請用快取加速相依安裝，並讓任何一步失敗就中止後續。先給我設定檔，我看過再啟用。
```

**為什麼有效：** 把 CI 的每個階段明確列出，AI 才不會漏掉關鍵步驟或順序錯誤。「失敗即中止」和「快取」是 CI 的基本要求，明寫出來確保產出的 pipeline 真的能用且不浪費時間。

### 3. 環境變數與密鑰管理

```
這個專案目前有些設定值寫死在程式碼裡：
[貼上相關程式碼]

請把它們改成從環境變數讀取：
- 建立 .env.example（只有 key、沒有真實值）
- 把 .env 加進 .gitignore
- 程式碼啟動時若缺少必要的環境變數要明確報錯

絕對不要把任何真實密鑰值寫進範例檔或程式碼。
```

**為什麼有效：** 寫死的設定和外洩的密鑰是部署兩大地雷。這個 prompt 把設定外部化、用 `.env.example` 當文件、缺值即報錯，一次解決可移植性與安全性，且明令不外洩真實值。

### 4. 部署前檢查清單

```
我準備把 [專案] 部署到 [平台，例如 Vercel / AWS]。

請先不要動手。給我一份「上線前檢查清單」，涵蓋：
- 環境變數是否都設好
- 資料庫 migration 是否需要先跑
- 有沒有寫死的本機網址 / port
- 錯誤監控 / log 是否就緒

逐項問我目前狀態，幫我確認都 OK 再進行。
```

**為什麼有效：** 部署事故大多來自「忘了某個設定」。用清單把容易遺漏的項目系統化地過一遍，把隱性的部署知識變成顯性的檢查步驟，大幅降低上線翻車機率。

### 5. 看懂部署錯誤

```
我的部署失敗了，這是 log：
[貼上完整部署 / build log]

請：
1. 找出真正導致失敗的那一行（不是後續的連鎖錯誤）
2. 解釋原因
3. 給我具體的修正步驟

如果資訊不足以判斷，告訴我還需要看哪個 log。
```

**為什麼有效：** 部署 log 又長又雜，新手常被一堆無關紅字嚇到。讓 AI 定位「真正的那一行」並過濾連鎖噪音，把問題收斂到根因，比自己瞎找快得多。

## 📝 寫文件

文件是 vibe coding 最容易跳過、之後最後悔的一環。好 prompt 讓 AI 從程式碼反推出準確的文件。

### 1. 從程式碼產生 README

```
請根據這個專案的實際程式碼，產生一份 README.md，包含：
- 一句話專案簡介
- 功能列表（依實際程式碼，不要編造沒有的功能）
- 安裝與啟動步驟（可複製貼上的指令）
- 基本使用範例

只描述程式碼裡真的存在的東西，不確定的地方標記出來問我。
```

**為什麼有效：** 強調「依實際程式碼、不要編造」能擋住 AI 幻想出不存在的功能。要求「不確定就標記」，讓你能補上它讀不出來的脈絡，產出一份真正準確的文件。

### 2. 寫函式 / API 註解

```
請為這段程式碼加上文件註解（[JSDoc / docstring / ...]）：
[貼上程式碼]

每個函式說明：
- 它做什麼（一句話）
- 參數的意義與型別
- 回傳值
- 會丟出什麼例外 / 錯誤

只加註解，不要改任何邏輯。
```

**為什麼有效：** 明確列出註解要涵蓋的面向（用途、參數、回傳、例外），避免 AI 寫出「// 這是一個函式」這種廢話註解。「只加註解不改邏輯」的限制讓這個動作安全無風險。

### 3. 寫使用範例

```
請為 [這個函式 / API / 套件] 寫使用範例。

要求：
- 從最簡單的「最小可用範例」開始
- 再給 2-3 個常見情境的範例
- 每個範例都是完整、可直接執行的，不要省略 import
- 範例旁用註解說明預期輸出

範例要真的能跑，不要寫虛構的用法。
```

**為什麼有效：** 「完整可執行、不省略 import、標註預期輸出」這幾條，把範例從「看起來像那樣」變成「真的能跑」。從最小範例漸進到常見情境，符合讀者的學習路徑。

### 4. 寫 commit message

```
這是我這次的改動：
[貼上 git diff 或描述改了什麼]

請依照 Conventional Commits 格式幫我寫 commit message：
- 標題：type(scope): 簡短描述（祈使句、不超過 50 字元）
- 內文：說明「為什麼」改，而不只是「改了什麼」

type 用 feat / fix / refactor / docs / test / chore 其中之一。
```

**為什麼有效：** 給定明確格式（Conventional Commits）和「說明為什麼而非什麼」的原則，讓 commit 訊息變成有用的歷史紀錄。標題長度與 type 限制讓訊息整齊一致，方便日後查閱與自動產生 changelog。

### 5. 寫架構說明文件

```
請為這個專案寫一份架構說明（ARCHITECTURE.md），幫助新加入的人快速理解。

請先讀過主要的程式碼，然後說明：
- 整體架構與資料流（從請求進來到回應出去）
- 主要模組各自的職責
- 重要的設計決策與為什麼這樣設計

用文字 + 簡單流程圖（mermaid 或 ASCII）說明，鎖定「大圖」，不要逐行解釋。
```

**為什麼有效：** 要求 AI「先讀程式碼再說明資料流與職責」，產出的是真正反映現況的架構圖，而非通用模板。聚焦「大圖與設計決策」正是新人最需要、卻最難從程式碼自行拼湊的部分。

## 🧠 Prompt 通用心法

這些不是特定情境的模板，而是讓「任何 prompt 都更有效」的通用技巧，可疊加到上面所有分類。

### 1. 給角色與標準

```
你現在是一位資深的 [領域，例如：後端工程師 / 資安專家]，
有 10 年實務經驗，重視 [可維護性 / 安全性 / 效能]。

請以這個身分和標準來處理我接下來的需求：[需求]。
```

**為什麼有效：** 指定專業角色與價值取向，能把 AI 的回答錨定到對應的專業水準與關注點。「資深資安專家」會主動想到邊界與漏洞，「重視可維護性的工程師」會避免炫技寫法——角色設定直接形塑產出品質。

### 2. 要它先問問題

```
在開始做 [任務] 之前，請先問我 3-5 個你需要釐清的問題，
這些問題的答案會影響你怎麼做。

等我回答完，你再開始。
```

**為什麼有效：** 主動要求 AI 提問，把它腦中那些「擅自假設」攤開來讓你確認。很多返工源自一開始的誤解，前置的幾個問題就能把方向校準，省下後面大量來回。

### 3. 要它解釋推理

```
請處理這個需求：[需求]。

在給出最終答案前，先一步步說明你的思考過程和為什麼這樣選擇。
如果有多種做法，列出來比較取捨，再告訴我你推薦哪個。
```

**為什麼有效：** 要求「先推理再給答案」會顯著提升複雜問題的正確率，也讓你看得到它的判斷依據。當它攤開取捨時，你能在它做決定前介入修正方向，而不是被動接受結果。

### 4. 給範例（少樣本）

```
我想要 [產出類型，例如：API 回應格式 / commit 訊息 / 命名]。

這是我喜歡的風格範例：
[範例 1]
[範例 2]

請照同樣的風格產生 [數量] 個 / 處理我接下來的內容。
```

**為什麼有效：** 與其用文字描述你要的風格（容易詞不達意），不如直接給範例。AI 從具體樣本中抓取模式的能力很強，給 2-3 個範例往往比一整段形容詞更精準地傳達你的期望。

### 5. 迭代而非一次到位

```
我們用迭代的方式做 [任務]。

先給我一個粗略的第一版（不用完美），我會給回饋，
我們一輪一輪改進。每一輪你只根據我的回饋調整，不要大改其他部分。
```

**為什麼有效：** 複雜需求一次說清楚幾乎不可能。明白宣告「迭代模式」並要求「每輪只改回饋處」，能避免 AI 每次都推倒重來，讓成果穩定地朝你要的方向收斂，而不是反覆橫跳。

## 🛡️ 護欄 Prompt（不讓 AI 亂動）

這一類專門用來「上鎖」，防止 AI 自作主張、越界改動。可以單獨用，也可以塞進 CLAUDE.md。

### 1. 規劃確認閘門

```
從現在開始，任何修改程式碼的動作之前，你都必須：
1. 先說明你打算做什麼、會動哪些檔案
2. 等我明確回覆「執行」或「OK」後才動手

在我同意之前，只討論、不修改。
```

**為什麼有效：** 在「想法」和「動手」之間插入一個人為確認閘門。這是最有效的護欄——讓你在任何改動落地前都有否決權，徹底杜絕「我還沒說完它就改完了」。

### 2. 凍結檔案範圍

```
這次任務只允許修改這些檔案：
[列出檔案清單]

其他任何檔案都不准碰。如果你認為需要改清單以外的檔案，
先停下來告訴我為什麼，由我決定，不要自己動。
```

**為什麼有效：** 明確的白名單把 AI 的「作用範圍」鎖死。它最危險的行為是「順手改相關檔案」，而檔案級的硬邊界能直接掐掉這種範圍蔓延。

### 3. 最小改動原則

```
請用「最小改動」解決這個問題：
- 只改真正必要的那幾行
- 不要重構、不要改格式、不要重新命名、不要動不相關的程式碼
- 不要「順便」優化任何東西

如果你很想改別的地方，先列出來問我，不要直接動。
```

**為什麼有效：** AI 天生愛「順便優化」，結果一個小修復變成上百行的 diff，難以 review 也容易夾帶新 bug。強調最小改動讓 diff 保持精簡、聚焦、可信。

### 4. 禁止刪除與破壞

```
規則：
- 不要刪除任何現有的測試、註解或功能，除非我明確要求
- 不要改動現有函式的對外介面（參數、回傳值）
- 如果你的修改可能破壞現有行為，先警告我

遵守這些規則的前提下進行。
```

**為什麼有效：** 明確列出「不可破壞的資產」（測試、介面、現有功能），保護那些 AI 容易為了達成目標而犧牲的東西。把保護對象講清楚，比事後發現它刪了你的測試好得多。

### 5. 不懂就問，不要亂猜

```
在這次任務中，如果遇到任何不確定的地方
（需求不明、有多種做法、缺少資訊、要做重要技術選擇），
請「停下來問我」，不要自己假設一個答案繼續做。

我寧願多回答幾個問題，也不要你猜錯方向做一大堆。
```

**為什麼有效：** AI 預設會「填補空白」——猜一個答案然後一路做下去，方向錯了就是大量返工。明確授權它「可以停下來問」，把錯誤擋在發生之前，比事後重做省太多。

## 📋 CLAUDE.md 模板

`CLAUDE.md`（或 Cursor 的 `.cursorrules`）是放在專案根目錄的「常駐指令」，AI 每次對話都會讀。把上面的護欄寫進去，就不用每次重貼。

> 💡 完整可直接使用的範本請見 **[CLAUDE-md-template.md](./CLAUDE-md-template.md)**。以下是幾個可單獨取用的區塊片段。

### 1. 專案概覽區塊

```markdown
## 專案概覽
- 這是什麼：[一句話]
- 技術棧：[語言 / 框架 / 資料庫]
- 進入點：[主要檔案，例如 src/main.ts]
- 啟動方式：`[指令]`
- 測試方式：`[指令]`
```

**為什麼有效：** 讓 AI 一開始就掌握全局，不用每次靠猜或亂翻檔案。明確的啟動 / 測試指令也讓它在改完後能自我驗證。

### 2. 程式規範區塊

```markdown
## 程式規範
- 語言版本：[x.x]，不要用已棄用語法
- 風格：遵循 [ESLint / Prettier / Black] 設定
- 命名：[camelCase / snake_case ...]
- 新程式碼請沿用專案現有的寫法與結構
```

**為什麼有效：** 把風格約束變成常駐規則，避免每個新檔案都長得不一樣。「沿用現有寫法」這條能讓產出自然融入既有程式碼。

### 3. 護欄區塊（最重要）

```markdown
## 工作規則（請嚴格遵守）
- 修改程式碼前，先說明計畫並等我確認
- 採最小改動原則，不要順便重構或優化
- 不要刪除既有測試 / 註解 / 功能
- 不確定時停下來問我，不要自行假設
- 不要改動現有函式的對外介面
```

**為什麼有效：** 把最有效的幾條護欄常駐化，等於對每一次對話都自動上鎖，不必每輪重貼。這是 CLAUDE.md 最有價值的部分。

### 4. 禁止事項區塊

```markdown
## 不要做的事
- 不要安裝我沒同意的新套件
- 不要修改 [設定檔 / migration / CI 設定]，除非我明確要求
- 不要把密鑰 / API key 寫進程式碼
- 不要產生大量無關的「順便改善」
```

**為什麼有效：** 用黑名單明確標出高風險動作。負面表列（「不要做 X」）常常比正面指示更能有效約束 AI 的越界行為。

### 5. 上下文指引區塊

```markdown
## 給 AI 的提示
- 重要的商業邏輯在：[路徑]
- 這些檔案很敏感，改動前務必三思：[清單]
- 常見陷阱：[專案特有的坑，例如：時區一律用 UTC]
- 相關文件：[連結]
```

**為什麼有效：** 把專案特有的「只有老手才知道的眉角」寫下來，等於給 AI 一份新人交接文件，大幅減少它踩專案專屬地雷的機率。

## 🔧 工具對應表

不同工具各有強項，用對場景事半功倍。

| 工具 | 最適合的情境 | 強項 | 注意 |
|------|------------|------|------|
| **Lovable** | 從零做一個有畫面的 Web App、做 MVP、給非工程師快速生原型 | 全流程生成（前端 + 後端 + 資料庫），所見即所得，部署方便 | 客製化深度有限，複雜邏輯仍需手動 |
| **Bolt** | 快速把一個想法變成可跑的全端 prototype、在瀏覽器裡即時預覽 | 瀏覽器內即時環境、生成速度快、適合 demo | 大型專案維護性較弱，較適合短週期 |
| **Cursor** | 在既有 codebase 上開發、需要在 IDE 裡精準控制改動 | 編輯器內 AI、能讀整個專案脈絡、`.cursorrules` 護欄、適合日常開發 | 需要自己懂程式，吃使用者的 prompt 功力 |
| **Claude Code** | 終端機內的代理式開發、多檔案重構、跑指令 + 測試 + 除錯的自動化流程 | 能讀寫檔案、執行指令、`CLAUDE.md` 常駐規則、適合複雜多步驟任務 | CLI 操作，學習曲線稍高 |

**選擇心法：**

- 🆕 **完全沒有 code、想快速看到成品** → Lovable / Bolt
- 🛠️ **已經有專案、要在裡面改東西** → Cursor / Claude Code
- 🧩 **多檔案、多步驟、要跑測試和指令** → Claude Code
- 👀 **想邊看畫面邊調** → Lovable / Bolt / Cursor

## 🇬🇧 English Summary

**vibe-coding-prompts** is a curated collection of copy-paste-ready prompt templates for *vibe coding* (building software with AI assistants). Every prompt comes with a **"why it works"** explanation, so you understand the principle and can adapt it to your own needs.

It solves three common pain points when coding with AI:

1. **Not knowing how to ask** — vague prompts produce unusable output.
2. **The AI goes rogue** — you ask for one button, it refactors half the project.
3. **Unpredictable results** — sometimes one shot, sometimes ten rounds of back-and-forth.

**Categories** (each with 5+ templates):

- 🚀 Project Initialization
- ✨ Adding Features
- 🐛 Debugging
- ♻️ Refactoring
- 🧪 Writing Tests
- 🎨 UI / Styling
- 🗄️ Database / SQL
- 🚢 Deployment / DevOps
- 📝 Writing Docs
- 🧠 General Prompting Principles
- 🛡️ Guardrail Prompts (keep the AI from going off-script)
- 📋 CLAUDE.md Templates

It also includes a **tool comparison table** (Lovable / Bolt / Cursor / Claude Code — what each is best for) and a ready-to-use **[CLAUDE.md template](./CLAUDE-md-template.md)**.

**Contributions welcome** — see **[CONTRIBUTING.md](./CONTRIBUTING.md)**. The one rule: quality over quantity. Every prompt must be copy-paste ready and explain *why* it works.
