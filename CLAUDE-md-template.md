# 📋 CLAUDE.md 完整範本

這是一份**可以直接複製到你專案根目錄**的 `CLAUDE.md` 範本。

`CLAUDE.md` 是放在專案根目錄的常駐指令檔，AI 編碼工具（Claude Code、Cursor 等）每次對話都會自動讀取。把專案資訊和工作規則寫進去，就不用每次重貼，也能大幅減少 AI 亂動、亂猜的情況。

---

## 🚀 怎麼用

1. 複製下方 **「✂️ 範本本體」** 區塊的全部內容
2. 在你的專案根目錄建立一個檔案叫 `CLAUDE.md`（Cursor 使用者可命名為 `.cursorrules`）
3. 把 `[方括號]` 裡的內容換成你專案的實際資訊
4. 用不到的區塊可以刪掉——**少而精準**比長而空泛更有效

> 💡 原則：CLAUDE.md 不是越長越好。AI 每次都要讀它，太冗長反而稀釋重點。只留下「真的會影響它行為」的內容。

---

## ✂️ 範本本體（複製這段）

```markdown
# CLAUDE.md

> 給 AI 編碼助手的常駐指令。請在每次協助前閱讀並遵守。

## 專案概覽
- 這是什麼：[一句話描述這個專案在做什麼]
- 技術棧：[語言 / 框架 / 主要套件 / 資料庫]
- 進入點：[主程式檔案，例如 src/main.ts]
- 啟動方式：`[啟動指令，例如 npm run dev]`
- 測試方式：`[測試指令，例如 npm test]`
- 建置方式：`[建置指令，例如 npm run build]`

## 程式規範
- 語言版本：[例如 Node 20 / Python 3.12]，不要使用已棄用（deprecated）的語法或 API
- 程式風格：遵循專案的 [ESLint / Prettier / Black / ...] 設定
- 命名慣例：[camelCase / snake_case / PascalCase ...]
- 新增程式碼時，請沿用專案中現有的寫法、結構與資料夾位置
- 註解語言：[繁體中文 / English]

## 工作規則（請嚴格遵守）
- 修改任何程式碼之前，先說明你的計畫（要動哪些檔案、改什麼），等我確認後再動手
- 採用「最小改動」原則：只改必要的部分，不要順便重構、改格式或重新命名
- 不要刪除既有的測試、註解或功能，除非我明確要求
- 不要改動現有函式的對外介面（參數、回傳值），除非我同意
- 遇到不確定的地方（需求不明、有多種做法、要做重要技術選擇）時，停下來問我，不要自行假設

## 不要做的事
- 不要安裝我沒有同意的新套件 / 相依
- 不要修改 [設定檔 / migration / CI / 環境變數] 等敏感檔案，除非我明確要求
- 不要把密鑰、API key、密碼等寫進程式碼或 commit
- 不要產生大量無關的「順便優化」，讓我的 diff 保持小而聚焦

## 專案特有提示
- 重要的商業邏輯位於：[路徑]
- 這些檔案很敏感，改動前請三思：[檔案清單]
- 常見陷阱：[專案特有的坑，例如：所有時間一律存 UTC / 金額用整數存「分」]
- 相關文件：[文件連結，若有]

## 溝通方式
- 回覆用：[繁體中文 / English]
- 我的程度：[初學者 / 中級 / 資深]，請用對應的深度說明
- 改完程式碼後，請簡短總結你做了什麼、為什麼，以及我該怎麼驗證
```

## 🧩 各區塊說明（為什麼要有這段）

### 專案概覽
讓 AI 一開始就掌握全局，不必每次靠猜或亂翻檔案找線索。特別是**啟動 / 測試 / 建置指令**——有了它們，AI 改完程式碼後能自己跑起來驗證，而不是丟給你一份它沒驗證過的程式碼。

### 程式規範
把風格約束變成常駐規則，避免每個新檔案風格都不一樣、新舊語法混用。**「沿用現有寫法」**這條尤其重要，能讓 AI 的產出自然融入既有程式碼，而不是各寫各的。

### 工作規則（最重要的護欄）
這是整份檔案最有價值的部分。把最有效的幾條護欄常駐化——**先計畫再動手、最小改動、不刪東西、不確定就問**——等於對每一次對話都自動上鎖，不必每輪重貼。AI 失控的行為大多能被這幾條擋下來。

### 不要做的事
用**負面表列**標出高風險動作（裝套件、改設定、寫死密鑰、過度優化）。實務上「不要做 X」常常比正面指示更能有效約束 AI 的越界行為。

### 專案特有提示
把那些「只有老手才知道的眉角」寫下來，等於給 AI 一份新人交接文件。每個專案都有自己的地雷（時區、金額單位、敏感檔案），明寫出來能大幅降低踩雷機率。

### 溝通方式
指定回覆語言和你的技術程度，讓 AI 的說明深度剛好對得上你；要求它**改完做總結**，讓你不用逐行讀 diff 也能快速掌握改了什麼、該怎麼驗證。

## 📏 精簡版（小專案 / 快速原型用）

如果是小專案或臨時原型，不需要完整版，這個精簡版就夠：

```markdown
# CLAUDE.md

## 專案
- [一句話描述]，技術棧：[...]
- 啟動：`[指令]`　測試：`[指令]`

## 規則
- 改 code 前先講計畫，等我確認再動手
- 最小改動，不要順便重構或優化
- 不刪既有測試 / 功能
- 不確定就問，不要自己猜
- 不裝我沒同意的套件

## 溝通
- 用 [繁體中文 / English] 回覆，改完做簡短總結
```

**為什麼有效：** 即使是最精簡的版本，只要保住「先計畫、最小改動、不刪東西、不確定就問」這四條核心護欄，就能擋掉大部分 AI 亂動的情況。護欄的價值不在長度，而在有沒有講到關鍵點。
